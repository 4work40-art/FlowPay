# Глобальное рыночное исследование — сверка с AUDIT_2026.md и AUDIT_2026_USER_PANEL.md

Источник — открытые профильные ресурсы (Baymard Institute, Nielsen Norman
Group, сравнения Bill.com/Melio/QuickBooks, данные по WhatsApp Business API
2026). Задача: проверить, не противоречат ли предыдущие два аудита мировой
практике, и найти то, что мы пропустили, глядя только на РФ-рынок.

## Что подтверждает мировая практика

- **Baymard**: каждый лишний тап до ключевой функции снижает завершение
  задачи на 5–10%. **Подтверждает приоритет PAN-4 (мобильный аудит)** и
  UX-1 (дашборд — один главный показатель) из первого аудита — эта задача
  так и не была реализована, поднимаю её значимость.
- **Baymard**: 70% пользователей бросают онбординг, если он занимает больше
  времени, чем ожидалось. **Подтверждает UX-3 (онбординг-чеклист)** — тоже
  ещё не реализован. Добавляю измеримую цель: путь от регистрации до
  первого счёта на контроле — не дольше 3 минут.
- **Nielsen Norman**: визуальная перегруженность — главная причина отказа в
  финтех-интерфейсах; дашборд не должен выглядеть как таблица CFO.
  Совпадает с находкой UX-1 из первого аудита один в один — глобальные
  данные добавляют вес, не меняют суть.
- **Progressive disclosure** — тренд 2026 везде (SaaS, enterprise, финтех):
  скрывать сложность, раскрывать по запросу. Прямая инструкция к тому, как
  делать UX-1/PAN-5 (статусы счёта, тултипы), а не абстрактный совет.

## Что мировая практика показала нового (не было ни в одном из двух прошлых аудитов)

### GLB-1: Публичная ссылка на счёт для контрагента (customer-facing portal)
И Bill.com, и Melio дают получателю счёта (не пользователю сервиса) ссылку
для просмотра статуса и оплаты без регистрации в системе. У нас счёт видит
только тот, кто его завёл в своём кабинете — контрагент физически не может
посмотреть «оплачен ли я, сколько должен» без звонка/сообщения владельцу
счёта. Для сценария частичных оплат (наша ключевая боль по ТЗ) это особенно
критично: обе стороны сделки должны видеть один и тот же остаток долга, а
не только тот, кто завёл счёт в системе.
→ Это не мелочь, это структурная возможность, отсутствующая в продукте.

### GLB-2: Notification-слой должен быть канало-агностичным, не Telegram-специфичным
WhatsApp Business API в 2026 резко подешевел для utility-сообщений
(напоминания об оплате — ровно наш кейс) в развивающихся странах. Для РФ
сейчас правильнее Telegram (панель это подтвердила), но если проект когда-
либо выйдет за пределы РФ или частично работает с зарубежными
контрагентами — стоит проектировать Notification Module как абстракцию
"канал + шаблон", а не жёстко зашивать Telegram Bot API в бизнес-логику.
→ Это инженерная рекомендация к реализации PAN-2, не новая фича для
пользователя.

## Обновлённый сводный приоритет (все три документа вместе)

Считаю, что после этого исследования разумный порядок такой:

1. **GLB-1 (публичная ссылка на счёт)** — структурная возможность, которой
   вообще нет, и она напрямую бьёт в главную боль продукта («обе стороны
   видят один остаток долга»). Более фундаментально, чем любая из PAN-задач.
2. **PAN-1 + PAN-2 (SMS-вход + Telegram-уведомления, с GLB-2 как архитектурным
   принципом)** — уже согласовывали как следующий шаг.
3. **UX-1/UX-3 (дашборд с одним главным показателем + онбординг с целью
   «до 3 минут»)** — так и не сделаны с первого аудита, теперь дважды
   подтверждены (внутренним аудитом и мировыми данными).
4. Остальное (PAN-3 импорт выписки, PAN-4 мобильный аудит, PAN-5..8,
   MKT-1..3) — без изменений в приоритете.
