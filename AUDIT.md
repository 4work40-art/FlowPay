# Аудит проекта и план рефакторинга — Счёт&Контроль

Согласовано с founder 2026-07-12. Этот документ фиксирует решения — все дальнейшие
работы по рефакторингу опираются на него, наравне с `foundation_invoice_control_platform`.

## 1. Расхождения с фундаментом — зафиксированные решения

| Вопрос | Было в документах | Решение |
|---|---|---|
| Free-тариф | 10 счетов (фундамент, `init.sql`) | **5 счетов** — обновить `invoice_limit` default и seed |
| Архитектура | Монолит (фундамент) vs микросервисы (AI-архитектура doc) | **Модульный монолит**. Микросервисы — не раньше реального роста нагрузки/команды |
| Роли пользователей | `owner/accountant/vendor_admin/readonly` в БД, `Owner/Accountant` в фундаменте | Оставить как в БД, `vendor_admin`/`readonly` — не MVP, не показывать в UI до отдельного решения |

## 2. Текущее состояние кода vs целевая архитектура

- `services/api-gateway/src/server.js` — 718 строк, все роуты (auth/invoices/payments/
  counterparties/admin) в одном файле. Нет модульного разделения, заявленного в
  фундаменте (Auth/Organization/Invoice/Payment/Document/Dashboard/Notification/
  Subscription/Vendor Admin/Analytics).
- **Multi-tenancy не реализован** — все данные жёстко привязаны к одной demo-организации
  (`ROADMAP.md`, Этап 3). Самостоятельная регистрация организации/владельца отсутствует.
- **Биллинга/ЮKassa нет** — только enum `plan_type` в схеме, ни таблиц транзакций,
  ни webhook-обработчика.
- **Document Module / распознавание счетов отсутствует** — только ручной ввод.
- **Notification Module отсутствует**.
- HTTPS/домен не настроены — блокер для приёма платежей через ЮKassa (эквайринг требует
  HTTPS и легальный домен).

Вывод: текущий UI-прототип (Dashboard, admin-панель вендора) опережает бэкенд —
показывает функциональность, которой в API ещё нет.

## 3. Экспертные роли для ревью (не отдельный штат, а перспективы при работе)

Полная оргструктура из "Product & Trust Council" (12+ ролей) избыточна для текущей
стадии — сама же выбранная архитектура предостерегает от overengineering. Вместо неё
на каждом шаге рефакторинга применяются 4 фильтра:

1. **Product/UX (МСП-фокус)** — сценарий "дропнул счёт → увидел остаток" остаётся
   минимальным по действиям; freemium-лимит не создаёт cognitive overload.
2. **Backend/Architecture** — модульность без преждевременного дробления на сервисы.
3. **Billing/Compliance (ЮKassa + РФ, 54-ФЗ)** — корректность вебхуков, чеков, статусов
   подписки.
4. **Security/Multi-tenancy** — жёсткая изоляция данных между организациями на уровне
   каждого запроса (org_id scoping), аудит-логи (уже частично есть).

## 4. Порядок работ (согласован)

1. **Multi-tenancy + регистрация организаций** — форма регистрации организации и
   владельца, изоляция данных по `org_id` во всех запросах, применение лимита `free`
   тарифа (5 счетов) при создании счетов сверх лимита.
2. **Модуляризация backend** — разбить `server.js` на модули
   (`auth`, `organizations`, `invoices`, `payments`, `counterparties`, `admin`)
   без изменения внешнего API-контракта.
3. **Биллинг / ЮKassa** — таблица подписок и транзакций, создание платежа, webhook
   подтверждения оплаты, апгрейд/даунгрейд тарифа.

Каждый последующий этап (Document Module/OCR, Notification Module) — по факту
необходимости, не заранее (принцип уже зафиксирован в `ROADMAP.md`).
