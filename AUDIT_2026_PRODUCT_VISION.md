# Аудит и доработки — контроль оплат, календарь, товары, поставщики

Дата: июль 2026. Команда: продакт-исследователь рынка, архитектор данных,
3 разработчика (параллельно, каждый — своя атомарная зона), QA-интеграция.

## 0. Зафиксированная позиция продукта (от владельца)

FlowPay даёт каждому пользователю:
1. Быстрый контроль статуса оплаты по счетам от контрагентов.
2. Напоминания о платеже.
3. Календарь оплат.
4. **Не банковский сервис** — деньги через нас не идут, только учёт фактов.
5. Учёт купленных товаров/услуг: количество, динамика цены за штуку,
   оборачиваемость, сезонность.
6. Учёт поставщиков: рейтинг по объёму закупок.

Это сужает и уточняет курс: мы — инструмент **финансового и товарного
контроля закупок**, не бухгалтерия и не WMS/склад.

## 1. Рынок: как это делают в РФ

- **Платёжный календарь** — базовая функция почти всех учётных сервисов
  (МойСклад предлагает готовый шаблон плана/факта платежей); везде это
  таблица/грид по датам с суммами план/факт.
- **Напоминания** — Контур.Эльба заранее уведомляет о сроках (email/push),
  это ожидаемый минимум, а не преимущество.
- **Рейтинг/классификация поставщиков по объёму закупок** — стандартный
  ABC-анализ по правилу Парето: группа A — поставщики, дающие первые 80%
  суммарных закупок; B — следующие 15% (до 95%); C — оставшиеся 5%.
  Используется повсеместно в закупочной аналитике (ERP/SRM-практика).
- **Оборачиваемость и сезонность товаров** — в полноценных ERP считается
  через остатки склада (COGS/средний остаток). У нас склада нет и не будет
  (мы не WMS) — честная адаптация: считаем **частоту и объём закупки
  товара по месяцам** (сколько раз и на какую сумму куплено) — это
  реальный, доступный без остатков прокси того же вопроса «что покупаем
  часто/редко и когда», и сезонность видна как раз по месячной динамике.

## 2. Аудит текущего состояния

| Область | Есть | Не хватает |
|---|---|---|
| Статус оплаты счёта | Полный жизненный цикл статусов, дашборд | — |
| Напоминания | Только «дайджест просрочки» (после срока) | Проактивное напоминание **до** наступления срока — самое ценное для «не пропустить платёж» |
| Календарь оплат | Есть таблица счетов с сортировкой по сроку | Нет визуального календаря (сетка по датам) |
| Товары/услуги в счёте | Счёт хранится одной суммой | Нет позиций (наименование/кол-во/цена) вообще — без этого невозможны ни динамика цены, ни сезонность |
| Рейтинг поставщиков | Есть сумма долга по контрагенту | Нет ранжирования по объёму закупок (ABC) |
| Дубли платёжных поручений | Не было проверки при ручном/повторном вводе | **Исправлено в этом релизе** |

## 3. Разбивка на атомарные задачи и распределение

| № | Задача | Кто | Зависимости |
|---|---|---|---|
| 1 | Защита от дублей платёжного поручения (распознавание + серверная проверка) | Я | — |
| 2 | Схема позиций счёта (`invoice_items`) + CRUD при создании/просмотре счёта | Я (фундамент, зона высокого риска конфликтов) | — |
| 3 | Календарь оплат + напоминания за N дней до срока (email + видимость в интерфейсе) | Агент-бэкенд+фронтенд A | — |
| 4 | Рейтинг поставщиков по объёму закупок (ABC) | Агент-бэкенд+фронтенд B | — |
| 5 | Аналитика по товарам: динамика цены, частота/объём закупки по месяцам (сезонность) | Агент-бэкенд+фронтенд C | Задача 2 (схема) |

Задачи 3 и 4 не зависят от задачи 2 и друг от друга — выполняются
параллельно с задачей 2. Задача 5 стартует после готовности схемы (я
фиксирую контракт таблицы/API заранее, агент C работает по контракту).

## 4. Осознанно не берём в этот релиз

- Автоматическое извлечение позиций товаров из OCR/скана счёта — сейчас
  добавляем только ручной ввод позиций; авто-разбор таблицы товаров из
  документа — отдельная будущая задача (нужно расширять
  `documentRecognizer.js` построчным разбором таблицы, это большая
  отдельная работа).
- Пуш-уведомления в браузере/мобильные пуши — напоминания пока только по
  email (единственный согласованный канал, см. предыдущие решения по
  SMS/Telegram).
