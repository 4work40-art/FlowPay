# Распознавание документов — сравнение с рынком РФ и план доработок

Дата: июль 2026. Готовили: архитектор распознавания (я), с опорой на открытые
источники — см. ссылки в конце.

## 1. Как это устроено у банков РФ

| Решение | Формат на входе | Технология | Что автозаполняется |
|---|---|---|---|
| СберБизнес | PDF/JPG/JPEG/PNG, фото со смартфона | ABBYY Mobile Imaging SDK + ABBYY Mobile OCR Engine (улучшение фото → распознавание) | Из счёта в формате «1С» — сразу готовое платёжное поручение; пользователь проверяет и подписывает |
| Т-Бизнес | PDF, TXT (формат 1С), QR-код на счёте | Собственный парсер 1С-формата + OCR для сканов; отдельно — реестры (в т.ч. экспорт в Excel/CSV) | Платёжное поручение целиком; реестр операций выгружается в Excel |
| СБИС / Контур.Диадок | ЭДО (структурированный XML), а не «просто файл» | Между операторами ЭДО документы уже структурированы — распознавание нужно только на границе бумага→система | Все стандартные поля счёта-фактуры/УПД, т.к. документ изначально в машиночитаемом виде |

Общий паттерн у всех: **фото/скан → предобработка изображения → OCR/шаблон →
черновик → обязательная проверка и подтверждение пользователем**. Никто не
проводит документ в учёт без подтверждения — это ровно то, что уже сделано у
нас, и на этом принципе продолжаем.

Второй паттерн: банки отдельно умеют читать **машиночитаемые форматы** (1С,
QR с реквизитами) — там, где формат уже структурирован, OCR не нужен вообще,
данные читаются напрямую. Мы уже поддерживаем ближайший аналог — 1С-выгрузку
банковской выписки (`1CClientBankExchange`) для платежей. Логично сделать
то же для счетов (реестр в Excel — стандартный способ выгрузки счетов из 1С,
МойСклад, СБИС и подобных систем).

## 2. Открытые методы распознавания — что применимо у нас

Рынок открытого ПО для инвойсов в 2026 делится на три уровня:

1. **Шаблонный парсинг** (invoice2data и аналоги) — regex/YAML-шаблон под
   каждого поставщика. Хорошо работает при небольшом числе стабильных
   форматов; у нас поставщиков (контрагентов пользователя) потенциально
   тысячи разных — шаблон на каждого не масштабируется.
2. **OCR + эвристики** (Tesseract/PaddleOCR + regex по меткам полей) — то,
   что уже используем. При правильной предобработке изображения (выравнивание,
   контраст, бинаризация) точность Tesseract 5 (LSTM-движок) для
   машинописных бланков РФ (счета, платёжки) вполне достаточна.
3. **Vision-language модели** (Donut, LayoutLMv3, Qwen2.5-VL) — не требуют
   шаблонов, понимают произвольную вёрстку, но требуют GPU и/или дообучения.

**Решение: остаёмся на уровне 2, но улучшаем его.** Причины:
- Наш прод — VPS 1 CPU / 1 ГБ RAM (см. `SUMMARY.md`); GPU-модели там
  физически не запустить, а внешний GPU-API — это (а) деньги за токен,
  (б) отправка финансовых документов на сторонний сервер, что для «мы не
  банк, но работаем с деньгами клиентов» продукта — лишний риск и лишняя
  запись в политику конфиденциальности.
- Regex/эвристики **прозрачны и проверяемы**: для денег это осознанный
  выбор — если сумма извлечена неверно, видно, из какой строки и почему,
  и это ловится юнит-тестом. У чёрного ящика ML это не так.
- Ограничение уровня 2 — качество зависит от предобработки изображения и
  полноты словаря меток. Это и есть основной вектор улучшения ниже.

## 3. Перечень доработок (что добавляем)

### 3.1 Качество распознавания (движок)
- **Предобработка изображения перед OCR**: grayscale + нормализация контраста
  + резкость перед Tesseract — стандартный приём (используется и ABBYY
  Mobile Imaging). Библиотека `sharp` (де-факто стандарт для Node.js,
  привязка к libvips, никаких внешних сервисов).
- **Классификация документа через взвешенные признаки**, а не первый
  сработавший `if` — точнее отличает счёт / счёт-фактуру / УПД / платёжку,
  когда в тексте встречаются несколько ключевых слов сразу.
- **Оценка уверенности (confidence)** — доля полей, которые удалось найти,
  возвращается вместе с результатом, чтобы пользователь видел, чему
  доверять меньше.

### 3.2 Полнота полей по законодательству РФ
- **Платёжное поручение** (форма 0401060, Положение Банка России № 762-П,
  приложение 1) — расширяем со «номер/дата/сумма/назначение» до полного
  набора: плательщик и получатель (ИНН/КПП, наименование, р/с, банк, БИК,
  к/с), очередность платежа, вид операции, УИН, КБК/ОКТМО (для бюджетных
  платежей).
- **Счёт-фактура / УПД** (ст. 169 НК РФ, Постановление Правительства РФ
  № 1137) — новый распознаваемый тип документа: продавец/покупатель
  (наименование, адрес, ИНН/КПП), номер и дата, валюта, ставка и сумма НДС,
  итого с НДС.
- **Счёт на оплату** (свободная форма, но де-факто стандартный состав) —
  добавляем ОГРН, адрес и банковские реквизиты поставщика (р/с, банк, БИК,
  к/с) — то, что нужно для последующей оплаты, а не только для учёта.

### 3.3 Excel/CSV — реестр счетов
Стандартный способ массовой выгрузки счетов из 1С/МойСклад/СБИС — не
единичный PDF, а реестр (таблица) на десятки строк. Добавляем разбор
`.xlsx`/`.xls`/`.csv` реестра и групповое создание счетов «черновик → проверка
→ подтверждение», по аналогии с уже работающим импортом банковской выписки.

### 3.4 Хранение — без изменений принципа
Файл никогда не попадает на диск и в БД дольше времени обработки одного
запроса: PDF/изображения — во временную директорию, удаляемую сразу после
OCR; Excel — читается прямо из буфера в памяти, временных файлов не
создаётся вовсе. Ничего сверх этого добавлять не требуется — это уже было
верно для текущей реализации; ниже добавлен явный тест, фиксирующий
инвариант, чтобы будущие изменения его не сломали случайно.
