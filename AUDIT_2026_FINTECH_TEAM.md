# Аудит продукта «Счёт&Контроль» — сводный отчёт команды (июль 2026)

Состав виртуальной команды и зоны ответственности:

| Роль | Зона аудита |
|---|---|
| Архитектор финтех-систем | архитектура, целостность данных, масштабирование |
| Senior Backend (Node/Postgres) | код API, транзакции, биллинг, импорт выписок |
| Security-инженер | аутентификация, изоляция арендаторов, транспорт, секреты |
| Юрист-комплаенс (152-ФЗ, 54-ФЗ, 115-ФЗ, 242-ФЗ) | законодательство РФ |
| Продакт + UX/UI-дизайнер | соответствие рынку 2026, простота, mobile |

Проверено: все ветки (`main`, `claude/fintech-product-audit-dn8w8y` — расходятся
только двумя UI-коммитами редизайна), весь бэкенд (13 роутов, 10 lib-модулей),
схема БД и 8 миграций, фронтенд (29 страниц), docker-compose, deploy.sh,
пять предыдущих аудитов.

Позиционирование зафиксировано: **продукт — учёт и контроль счетов, оплат и
долга. НЕ банковское приложение, НЕ платёжный сервис.** Деньги контрагентов
через нас не проходят — мы только фиксируем факты. ЮKassa используется
исключительно для оплаты наших собственных подписок. Это принципиально
для комплаенса: лицензии ЦБ и требования 115-ФЗ к операторам платежей на нас
не распространяются, и в это положение не надо «дорастать» случайной фичей
(например, приёмом денег за счета клиентов через нашу кассу).

---

## 1. Что сделано хорошо (подтверждаем)

- Мультиарендная изоляция: `org_id` берётся из JWT, а не из тела запроса;
  выборочно проверили все роуты — межарендных утечек не нашли.
- Платежи: транзакция + `SELECT ... FOR UPDATE`, приведение BIGINT к числу,
  запрет переплаты, CHECK-констрейнты в БД — гонки и порча остатков закрыты.
- Пароли: bcrypt (pgcrypto), сброс пароля через хэшированные одноразовые
  токены, анти-oracle ответ на forgot-password, rate-limit на login/register.
- 152-ФЗ частично: обязательный чекбокс согласия с фиксацией времени
  (`pdn_consent_at`), страницы политики и оферты.
- Webhook ЮKassa не доверяет телу запроса — статус перепроверяется через API,
  есть идемпотентность по статусу транзакции.
- Аудит-журнал действий, автоперевод в OVERDUE фоновым джобом, автобэкапы БД.

## 2. Критические находки (блокеры, чинить до любого развития)

**C1. Продакшн работает по HTTP без TLS.** Пароли, JWT и все данные (включая
ПДн — email, имена, ИНН контрагентов) ходят открытым текстом. Это прямое
нарушение требований к защите ПДн при передаче (152-ФЗ, приказ ФСТЭК № 21)
и главный риск угона сессий. Решение известно и дешево: домен + Caddy/nginx
с Let's Encrypt перед стеком. Осознанная отсрочка «работаем по IP» дальше
недопустима — продукт публичный и принимает регистрации.

**C2. 54-ФЗ: продажа подписок без фискализации.** `createPayment` в ЮKassa
не передаёт объект `receipt` — кассовый чек покупателю подписки не
формируется. Для приёма оплаты от физлиц/ИП это нарушение 54-ФЗ. ЮKassa
умеет фискализацию «под ключ» — нужно передавать `receipt` (email плательщика,
позиция «Подписка …», ставка НДС по нашей системе налогообложения).

**C3. Оплаченный тариф никогда не истекает.** `current_period_end`
устанавливается (+30 дней), но ни один джоб её не проверяет: заплатив 990 ₽
один раз, организация остаётся на Pro навсегда. Нужен sweep (по аналогии с
overdueJob): истёк период → даунгрейд на free + email-уведомление за N дней.

**C4. Импорт банковской выписки: дубли и ложные сопоставления.**
- Повторная загрузка той же выписки создаёт платежи второй раз — нет
  дедупликации (нужен хэш строки выписки или уникальность
  дата+сумма+назначение в пределах org).
- Сопоставление `purpose.includes(inv.number)` — подстрока: счёт «1»
  «найдётся» в назначении платежа по счёту «104». Нужно сопоставление по
  границам слова/точному токену и режим «предложить, а не провести»
  (подтверждение оператором для неоднозначных строк).

**C5. Боевые доступы опубликованы в репозитории.** `SUMMARY.md` содержит IP
боевого сервера и логин/пароль demo-аккаунта. Историю git уже не очистить
дёшево — минимум: сменить пароль, убрать из файлов, впредь держать доступы
вне репозитория.

**C6. Токены: refresh-фикция и неполный отзыв.** `refresh_token` — это тот же
access-токен; 24-часовой JWT без ротации. Logout отзывает только текущий
`jti`; смена/сброс пароля не отзывает остальные активные сессии. Нужно:
короткий access (30–60 мин) + настоящий refresh с ротацией; при сбросе
пароля — отзыв всех сессий пользователя (например, `token_version` в БД).

## 3. Высокие риски (следующая очередь)

- **H1.** Rate-limiter «fail-open»: при недоступном Redis лимиты на login
  исчезают молча. Для auth-эндпоинтов оправдан in-memory fallback.
- **H2.** Публичная ссылка на счёт: невозможно отозвать, отдаёт всю историю
  платежей. Нужен флаг «публичный доступ вкл/выкл» на счёте и/или отзываемый
  токен ссылки.
- **H3.** ИНН/КПП не проверяются (ни контрольная сумма, ни длина 10/12).
  Для продукта про долги ИНН — ключевой реквизит.
- **H4.** Файлы-вложения лежат на локальном диске и **не попадают в бэкап**
  (`backup.sh` дампит только БД). Потеря диска = потеря документов.
- **H5.** CORS по умолчанию `*` (fallback в server.js) — на бою обязателен
  явный origin; убрать fallback, падать при пустом `CORS_ORIGIN`.
- **H6.** Ни одного автотеста и нет CI. Для финансовой логики (остатки,
  статусы, импорт) это уже риск регрессий — прошлый BIGINT-баг тому
  подтверждение. Минимум: интеграционные тесты платёжного цикла + GitHub
  Actions.
- **H7.** Импорт выписки открывает соединение из пула на каждую строку CSV —
  на большой выписке при пуле по умолчанию это деградация; одна транзакция
  на файл.

## 4. Комплаенс РФ — карта соответствия

| Норма | Статус | Что не хватает |
|---|---|---|
| 152-ФЗ (ПДн) | частично | отзыв согласия и удаление аккаунта с данными; уведомление РКН о начале обработки (реестр операторов); перечень третьих лиц (ЮKassa, SMTP-провайдер) в политике; шифрование канала (см. C1) |
| 242-ФЗ (локализация) | ок | хостинг TimeWeb (РФ) — зафиксировать в политике |
| 54-ФЗ (ККТ) | **нарушение** | фискальный чек при оплате подписки (C2) |
| 115-ФЗ (ПОД/ФТ) | не применяется | мы не оператор платежей; зафиксировать границу в оферте и не пересекать её продуктовыми фичами |
| 402-ФЗ (бухучёт) | ок | мы не первичка и не заменяем её; полезно указать в оферте, что сервис — управленческий учёт |

## 5. Соответствие рынку 2026 и план развития

Ядро («быстрый контроль счетов, оплат, долга») реализовано и работает.
Чего рынок РФ 2026 ждёт от такого инструмента и чего нет:

1. **Импорт 1CClientBankExchange (`*.txt`)** — стандартный формат выгрузки
   всех банков РФ (Сбер, Т-Банк, Альфа...). Наш CSV-импорт нестандартен;
   поддержка 1CClientBankExchange закроет 90 % реальных выписок. Формат
   открытый, парсер тривиален.
2. **Автозаполнение контрагента по ИНН** (DaData/ЕГРЮЛ API — бесплатный
   тариф достаточен): ввёл ИНН → название, КПП, адрес подтянулись. Убирает
   главную рутину и ошибки ввода.
3. **Email-напоминания о просрочке** — SMTP уже есть, канал согласован
   (SMS/Telegram исключены решением владельца). Дайджест «что просрочено
   сегодня» владельцу + опционально письмо контрагенту со ссылкой на
   публичную страницу счёта.
4. **QR СБП на публичной странице счёта** (статический QR по реквизитам,
   без приёма денег через нас — граница «не платёжный сервис» сохраняется).
5. **Повторяющиеся счета** (аренда, абонплата) — было отложено, пора.
6. **Показатель DSO / средний срок оплаты по контрагенту** на дашборде —
   одна цифра, отвечающая на вопрос «кто платит плохо».

### Предлагаемая последовательность релизов

- **R1 «Довериться можно» (1–2 недели):** C1–C6 целиком + H1, H3, H5.
  Ничего продуктового — только надёжность и закон.
- **R2 «Комплаенс-полнота» (1 неделя):** удаление аккаунта/отзыв согласия,
  обновление политики (третьи лица, локализация), уведомление РКН
  (организационное действие владельца), H2, H4.
- **R3 «Рынок 2026» (2–3 недели):** 1CClientBankExchange-импорт, ИНН-подсказки
  DaData, email-напоминания, DSO на дашборде.
- **R4 «Рост» (по метрикам):** повторяющиеся счета, QR СБП, тесты покрытием
  на всю финансовую логику + CI (начать закладывать уже в R1).

### Открытые библиотеки под задачи (проверенные, MIT/BSD)

- `csv-parse` — вместо самодельного CSV-парсера (кавычки, экранирование);
- `zod` — валидация тел запросов единообразно;
- `pino` — структурные логи вместо console.log;
- `node-cron` — расписания джобов (истечение подписок, дайджесты);
- DaData Suggestions API — подсказки по ИНН (это API, ключ бесплатный);
- Playwright + supertest — e2e и интеграционные тесты.

## 6. Статус исполнения (обновлено после релиза)

- **C1 (TLS/домен)** — отложено решением владельца; единственный открытый блокер.
- **C2–C6, H1–H7** — выполнено и проверено сквозными прогонами.
- **R2 (комплаенс)** — выполнено: удаление организации со всеми данными
  (настройки, только владелец, подтверждение паролем), политика дополнена
  третьими лицами (ЮKassa, SMTP, DaData) и правом на удаление.
  Организационно за владельцем: уведомление РКН, смена демо-пароля на бою.
- **R3 (рынок 2026)** — выполнено: импорт 1CClientBankExchange (включая
  windows-1251), автозаполнение контрагента по ИНН (DaData, требует
  DADATA_API_KEY), ежедневный email-дайджест просрочки (SMTP), средний срок
  оплаты (DSO) на дашборде, отключаемая публичная ссылка на счёт.
- **R4** — заложены тесты (node --test: ИНН, разбор выписок) и CI
  (GitHub Actions: синтаксис, тесты, tsc). Повторяющиеся счета и QR СБП —
  по метрикам использования, как и планировалось.

## 7. Итог

Продукт архитектурно здоров и делает то, что заявлено. Блокеры — не в
функциях, а в эксплуатации и законе: TLS, фискализация 54-ФЗ, вечные
подписки, дубли при импорте выписок, доступы в git. Всё это чинится за один
короткий релиз R1, после которого продукт можно спокойно показывать
платящим клиентам. Решения по R1–R4 — за владельцем.
