# Путь масштабирования — Счёт&Контроль

## Этап 0 — сейчас
- 1 VPS (TimeWeb Cloud), Docker Compose: Postgres + Redis + API + фронтенд
- Одна организация, реальная JWT-авторизация, доступ по HTTP + публичному IP
- Ручной деплой через `deploy.sh` (git pull + docker compose up -d --build)
- Ежедневный `pg_dump` в локальную папку на том же сервере

## Этап 1 — домен и HTTPS
- Купить/подключить домен, направить A-запись на IP сервера
- Поставить reverse-proxy (Caddy или nginx + certbot) перед API и фронтендом — автоматический Let's Encrypt сертификат
- Перевести `NEXT_PUBLIC_API_URL` / `CORS_ORIGIN` на `https://домен`
- Закрыть прямой доступ к портам 3000/3001 снаружи, пускать только через 443

## Этап 2 — команда внутри организации
- В БД уже есть роли (`owner`, `accountant`, `vendor_admin`, `readonly`) — не хватает только UI/API
- Добавить эндпоинт приглашения пользователя в организацию (email + временная ссылка/код)
- Экран управления пользователями (список, смена роли, деактивация)

## Этап 3 — самостоятельная регистрация организаций (multi-tenant)
- ✅ Форма регистрации новой организации + владельца (`POST /api/v1/auth/register`, страница `/register`)
- ✅ Free-тариф — 5 счетов (`invoice_limit` default), лимит уже блокирует создание счетов сверх лимита
- ✅ Модуляризация backend: `server.js` (718 строк) разбит на `lib/` (db, auth, http, audit) и `routes/` (auth, dashboard, invoices, payments, counterparties, users, audit, admin) — внешний API-контракт не менялся
- ✅ Биллинг через ЮKassa: `POST /api/v1/billing/checkout` создаёт платёж, `POST /api/v1/billing/webhook`
  подтверждает его (статус всегда перепроверяется через ЮKassa API, тело webhook не считается источником истины)
  и апгрейдит тариф + `invoice_limit` организации. Таблицы `subscriptions`, `billing_transactions`
  (`infra/postgres/migration_billing.sql`). Требует env `YOOKASSA_SHOP_ID` / `YOOKASSA_SECRET_KEY` — без них
  чекаут возвращает 503 `BILLING_NOT_CONFIGURED`, не ломая остальной API.

## Этап 4 — надёжность и эксплуатация
- Бэкапы БД — не только локально: выгрузка в S3-хранилище TimeWeb (offsite), проверка восстановления по расписанию
- Мониторинг: аптайм API/фронта, алерты в Telegram/почту при падении health-check
- CI/CD: авто-деплой при пуше в `main` (GitHub Actions → webhook на VPS), вместо ручного запуска `deploy.sh`
- Логирование ошибок (Sentry или аналог) вместо `console.error`

## Этап 5 — рост нагрузки
- Вынести Postgres в managed-БД (TimeWeb DBaaS) с отдельным бэкап-SLA — снимает риск потери данных при поломке VPS
- Реально задействовать Redis: кэш дашборда, rate-limiting, revocation-лист токенов (уже используется с Этапа 0 для logout)
- При росте RPS — второй инстанс API за балансировщиком, PgBouncer для пула соединений к БД
- Апгрейд тарифа VPS/DBaaS по факту нагрузки (сейчас 1 ГБ RAM — хватает на старте, но не более)

## Принцип
Каждый этап включается по факту необходимости — не делать заранее то, что не нужно прямо сейчас. Этап 1 (домен/HTTPS) — следующий логичный шаг, как только появится домен.
